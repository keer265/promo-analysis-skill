# READINESS_GATE

本文件定义 promo-analysis-skill 的“阶段闸门（readiness gate）”机制。

目标：
在真正开始分析前，先根据当前已经读取到的数据，判断 **能执行到哪一块**、**哪些模块会被阻塞**、**哪些可以降级执行**，然后必须先询问用户：

- 继续执行当前最高可执行版本
- 仅输出数据缺口与补充建议
- 停止执行

注意：
readiness gate 之前，必须先完成 `RUN_SETUP.md` 中的调用配置。

---

## 一、强制原则

1. **先补齐调用配置，再判级**
   不能在数据路径/来源、输出格式、输出路径未明确时直接进入分析。

2. **先判级，再分析**
   不能在未做可执行性检查时直接默认跑完整链路。

3. **必须显式列出“能做 / 不能做 / 降级做”**
   不要只说“数据不完整”，要具体到模块。

4. **必须给出下一步选项**
   在 readiness gate 之后，必须主动问用户：
   - 是否继续执行可做部分
   - 是否只输出缺口
   - 是否停止

5. **默认不强行继续**
   除非用户明确要求，否则在阶段闸门后不要擅自进入完整分析正文。

---

## 二、执行等级

### Level 0：不可执行
**典型情况**
- 缺订单表
- 日期无法解析
- 活动信息缺失且用户目标又明确要求活动评估
- 主键无法关联到可分析粒度

**可做**
- 仅输出缺口清单
- 给出需要补哪些表、哪些字段

**不可做**
- 任何正式活动效果评估
- 基线建模
- 增量识别

---

### Level 1：描述性版
**最低条件**
- 只有订单表，或只有可形成日级销售序列的数据

**可做**
- 数据概览
- 口径梳理
- 描述性统计
- 周期性初步识别
- 初步图表框架

**不能高质量做**
- 活动真实增量评估
- 反事实基线
- 提前消费 / 蚕食识别
- 预算优化

---

### Level 2：基础评估版
**最低条件**
- 订单表 + 活动表
- 日期可对齐
- 至少有一定非活动期样本

**可做**
- 数据检查
- 周期性识别
- 基础反事实基线
- 活动窗口增量评估
- 事件窗口动态
- 基础稳健性检查
- 初步商业建议

**降级做**
- 无曝光表时，不做高质量流量—转化—客单分解
- 无用户级稳定标识时，不做高质量用户分层
- 无 SKU 维度时，不做 SKU 蚕食

---

### Level 3：标准完整版
**最低条件**
- 订单表 + 活动表 + 曝光表
- 日期、SKU 粒度可对齐
- 非活动期样本足够

**可做**
- Level 2 的全部内容
- GMV 三因子拆解（View_UV × CVR × UValue）
- 折扣响应分析
- SKU 敏感度分析
- 更完整的提前消费 / 蚕食判断

**仍可能受限**
- 若缺预算字段，则预算优化只能基于实际优惠额近似
- 若缺用户画像增强字段，则用户分层偏行为型而非画像型

---

### Level 4：增强识别版
**最低条件**
- Level 3 条件满足
- 另有预算、活动类型、用户标签、渠道、实验分组或更细粒度活动单元

**可做**
- 更完整的 DID / 合成控制辅助验证
- 更细的用户分层
- 更可信的预算优化模拟
- 更具体的定向投放建议
- 更贴近落地执行的 A/B 方案

---

## 三、模块与字段依赖映射

| 模块 | 最低依赖 | 增强依赖 |
|---|---|---|
| 数据检查 | 任一结构化表 | 全部表 |
| 描述性统计 | 订单表 | 曝光表、类目、品牌 |
| 周期识别 | 订单表 + 日期 | 更长时间窗 |
| 基线构建 | 订单表 + 活动表 | 类目粒度、更多非活动期 |
| 增量评估 | 订单表 + 活动表 | 活动预算、活动类型 |
| 三因子拆解 | 订单表 + 曝光表 | 更稳定 UV 口径 |
| 用户分层 | user_id | 用户标签、渠道、新老客 |
| SKU 敏感度 | sku_id | 品牌、曝光 |
| 提前消费 / 蚕食 | 活动表 + 时间窗 + SKU/类目 | 更长前后窗口 |
| 预算优化 | discount_amount 或 budget | 活动单元级成本 |
| DID / 合成控制 | 活动表 + 足够对照期 | 更细分组字段 |

## 四、阶段闸门标准输出格式

在正式分析前，先输出以下结构：

### 1. 数据读取结果
- 已读取哪些表
- 每张表能识别到哪些关键字段
- 当前明显缺口是什么

### 2. 可执行等级
- 当前等级：Level X
- 判断原因：用 2~5 条简明说明

### 3. 当前可执行模块
- 列出能做的章节

### 4. 当前受阻模块
- 列出不能做或只能弱化做的章节

### 5. 降级方案
- 如果继续，准备采用什么降级分析路径

### 6. 用户选择
必须最后询问：

> 当前数据可以执行到 **Level X**。  
> 我可以：  
> 1. 继续执行可做部分；  
> 2. 只输出数据缺口与补充建议；  
> 3. 先停止在这里。  
> 你选哪一种？

---

## 五、推荐问法模板

### 模板 A：数据较完整
当前数据已满足 **Level 3（标准完整版）**。  
我可以继续完成周期识别、基线构建、活动增量评估、三因子拆解、用户/商品敏感度与预算优化；
但部分增强识别仍会受限于预算或用户标签字段。  
你要我：继续执行、仅输出缺口，还是先停止？

### 模板 B：数据中等完整
当前数据已满足 **Level 2（基础评估版）**。  
我可以继续做周期识别、基础基线与活动窗口增量评估；
但由于缺少曝光表/用户画像字段，三因子拆解和高质量用户分层会降级或跳过。  
你要我继续执行可做部分，还是只输出缺口清单？

### 模板 C：数据不足
当前数据只够到 **Level 1（描述性版）**。  
我可以先做数据概览、描述性统计和周期性初步识别；
但还不足以支撑正式的活动增量评估。  
你要我先做描述性版，还是只输出还需补充的数据项？
